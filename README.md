# Документація проєкту: Дашборд AI-Аналітики Amazon EU

### 1. Обраний технологічний стек та обґрунтування
* **Python + Streamlit:** Ідеальний стек для інтерактивних бізнес-додатків. Замість передачі сирого коду, нетехнічний користувач (sales-менеджер) отримує зручний веб-інтерфейс у браузері, де може просто перетягнути мишкою файли звітів (`march 2026.csv` та `april 2026.csv`) й миттєво побачити результат без роботи з консоллю.
* **Pandas:** Використано для автоматизації ETL-процесу (очищення даних). Скрипт самостійно усуває «бруд» у вивантаженнях Amazon: обробляє європейську нотацію (замінює коми на крапки у дробових числах, видаляє нерозривні пробіли тисяч `\xa0`) та динамічно відновлює ієрархію рядків, прив'язуючи кожен товар до свого маркетплейсу.
* **Plotly:** Забезпечує інтерактивну візуалізацію (менеджери можуть масштабувати графіки та наводити мишкою для деталізації).

### 2. Реалізована AI-фіча (Executive Summary)
* **Що зроблено:** Модуль автоматичного executive-аудиту портфеля. Він миттєво розраховує глобальні зміни метрик і людською мовою підсвічує аномалії бізнесу: лавиноподібне падіння прибутку в квітні на тлі меншого падіння продажів, вихід маркетплейсу Італії в чистий мінус, а також конкретні проблемні SKU (наприклад, лідера падіння `73-EFLH-T6DR`).
* **Чому саме вона:** Sales-менеджери витрачають години на ковиряння в таблицях із 250+ рядків. Штучний інтелект за 1 секунду фокусує увагу керівництва на критичних точках просідання (Action Items).

### 3. Логіка інкрементального імпорту даних (Як це було б реалізовано в продакшені)
Оскільки в реальному бізнесі звіти додаються щомісяця, для уникнення дублікатів та накопичення історії механізм працював би так:
1. При завантаженні нового CSV-файлу система формує унікальний складений ключ (Composite Key) для кожного рядка: `Місяць + Рік + Маркетплейс + SKU`.
2. Замість звичайного додавання (Insert), дані відправляються в базу даних за допомогою операції **UPSERT** (Update or Insert).
3. Якщо запис із таким ключем уже існує в базі (наприклад, менеджер повторно завантажує квітень), система оновлює фінансові метрики. Якщо ключа немає — додає новий рядок. Це гарантує 100% чистоту даних та виключає подвоєння продажів.

### 4. Масштабування системи при наявності більшої кількості часу
1. **Повний No-Code конвеєр (ETL):** Заміна ручного завантаження файлів на автоматичний збір даних за розкладом через ноду **n8n** напряму з **Amazon SP-API (Selling Partner API)**.
2. **Центральне сховище:** Збереження очищених даних у реляційну базу **Supabase (PostgreSQL)** для миттєвого завантаження дашборду без парсингу «важких» CSV «на льоту».
3. **Маркетинговий аналіз (PPC + P&L):** Автоматичне зведення рекламних звітів (`DE_*.csv` тощо) з фінансовими по ключу SKU для розрахунку показника **TACOS** (частка витрат на рекламу в загальному обороті) по кожній країні.
